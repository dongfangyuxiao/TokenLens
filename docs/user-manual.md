## 1. 快速开始

### 1.0 2026-03-31 更新要点

- 授权校验已**强制开启**，未导入有效 `license.json` 时所有扫描功能（投毒检测、增量审计、全量审计、即时分析）均被拦截，仅允许登录和查看系统设置
- 多模型联合校验全面落地：手动触发、计划任务、历史重跑均支持「审计模型 / 检查模型 / 验证模型」三段式链路配置
- 多模型共识结果自动沉淀为 Adaptive Skills，后续同语言扫描自动注入 Prompt 提升准确率
- 概览仪表盘统计卡片支持点击穿透至对应级别漏洞列表，漏洞统计趋势图同样支持点击跳转
- 漏洞列表新增独立紫色筛选栏，支持按严重级别和处理状态组合筛选
- 内置提示词新增 Java、PHP、Python、Go、Node.js、C#、Ruby、Rust、C/C++ 等语言专项安全技能

### 1.2 2026-03-30 更新要点

- AI 配置新增“启用多模型联合校验后自动优化 skills”勾选项
- 投毒检测、增量审计、全量审计支持分别配置审计模型、检查模型、验证模型
- 每个角色都可以单选或多选模型组成联合校验链
- 多模型共识结果会自动优化 skills，后续同类语言扫描会自动增强
- `majority / all` 下被多数模型否决的低可信模式会自动沉淀，用于压制误报噪音
- 内置新增 Java、PHP、Python、Go、Node.js、C#、Ruby、Rust、C/C++ 等常见语言专项提示词

### 1.3 2026-03-27 更新要点

- 即时分析（粘贴代码/上传文件）新增 AI 模型选择，可按模型配置执行
- 扫描状态新增实时进度百分比显示（顶部状态与概览同步）
- 扫描历史新增“实时漏洞”入口，扫描进行中可打开并自动刷新
- 应急排查结果支持分页与每页数量切换（20/50/100）
- 系统设置新增“配置文件检测”开关（默认关闭）
- Syslog Facility 支持多选，并改为中文可读说明标签
- 代码库配置页增加仓库列表；全量审计后自动清理失效仓库

### 1.4 2026-03-26 更新要点

- AI 配置统一为新版字段（`provider/model/api_key/base_url`），已移除旧版 DeepSeek/Anthropic Key 配置入口
- 支持本地模型（Ollama）与云端模型统一多模型管理
- 投毒检测、增量审计、全量审计均支持按扫描任务选择指定 AI 配置
- 新增应急排查页面：支持恶意依赖检测、全代码库关键词检索、组件清单查询
- 新增产品授权能力，支持机器码展示、授权文件导入、状态校验、到期提醒和功能点控制
- 界面已升级为深色科技安全控制台风格，默认首次进入为深色主题
- 恢复 Light / Dark 双主题切换，主要页面已补齐浅色模式样式
- 顶栏支持直接触发三类扫描，并在弹窗中选择本次使用的 AI 配置
- 扫描历史支持导出 HTML / PDF / JSON / Markdown / DOCX
- 管理员删除、空结果导出等边界行为已修复，提示信息更准确

### 1.5 访问平台

在浏览器中打开平台地址（默认 `http://localhost:8000`），进入登录页面。

### 1.6 首次登录

| 字段 | 默认值 |
|------|--------|
| 用户名 | `admin` |
| 密码 | `admin123` |

> **重要**：首次登录后请立即前往左侧 **账户管理** 修改默认密码。密码须包含大小写字母和数字，长度不少于 8 位。

```
① 添加代码平台账户  →  ② 配置 AI 模型  →  ③ 设置扫描计划  →  ④ 配置告警通知
```

## 3. 接入代码平台

进入 **代码平台** 页面，点击 **添加账户**。

### 3.1 GitHub

| 字段 | 说明 |
|------|------|
| 平台 | 选择 GitHub |
| 名称 | 任意显示名称 |
| Token | Personal Access Token，需要 `repo` 读权限 |
| Owner | GitHub 组织名或用户名（平台会拉取该 Owner 下所有可访问仓库） |

**如何创建 GitHub Token**：
1. 登录 GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
2. Generate new token，勾选 `repo` 权限
3. 复制生成的 Token 填入平台

---

### 3.2 GitLab（含腾讯工蜂 / 阿里云效 Codeup）

|------|------|

| Token | GitLab Personal Access Token，需要 `read_api` 权限 |
| Owner | 组织/群组名称 |
| 自定义域名 | 私有化部署时填写，如 `https://gitlab.your-company.com`；使用 GitLab.com 则留空 |

## 4. 配置 AI 模型

进入 **AI 配置** 页面。平台支持多个 LLM 配置档案，可为不同扫描计划分配不同模型。

### 4.1 添加 LLM 配置

点击 **新建配置**，填写：

| 字段 | 说明 |
|------|------|
| 名称 | 配置的显示名称，如"GPT-4o 生产"、"本地 Ollama" |
| 提供商 | 从下拉列表选择 |
| 模型 | 填写具体模型名称 |
| API Key | 对应提供商的 API 密钥 |
| 接口地址 | 使用官方地址时留空；私有化部署或代理时填写 |

### 4.2 各提供商参数参考

**OpenAI**
- 提供商：`openai`
- 模型：`gpt-4o` / `gpt-4o-mini` / `gpt-4-turbo`
- API Key：`sk-...`

**Anthropic Claude**
- 提供商：`anthropic`
- 模型：`claude-opus-4-6` / `claude-sonnet-4-6`
- API Key：`sk-ant-...`

**DeepSeek**
- 提供商：`deepseek`
- 模型：`deepseek-chat` / `deepseek-coder`
- API Key：DeepSeek 控制台获取
- 接口地址：`https://api.deepseek.com/v1`

**阿里通义千问**
- 提供商：`qwen`
- 模型：`qwen-max` / `qwen-plus`
- 接口地址：`https://dashscope.aliyuncs.com/compatible-mode/v1`

**本地 Ollama**
- 提供商：`ollama`
- 模型：`llama3` / `qwen2.5-coder` / `deepseek-coder`
- API Key：任意填写（Ollama 不校验）
- 接口地址：`http://localhost:11434/v1`

> 说明：建议优先使用“AI 配置列表”维护多模型，并在扫描触发弹窗/计划任务中按任务绑定具体模型。

### 4.3 多模型联合校验与自动优化 skills

在手动触发扫描或新增计划任务时，可以分别选择：

- `审计模型`
- `检查模型`
- `验证模型`

并可勾选：

- `根据审计结果优化对应 skills`

勾选后，本次联合校验会触发自动优化：

- 学习最终保留下来的真实漏洞模式
- 学习被检查/验证阶段否决的低可信模式
- 后续同任务、同语言文件扫描时，系统会自动把这些经验追加到已有 skills 中

适合的组合示例：

- `DeepSeek` 初审
- `GPT` 复核
- `Claude` 验证

### 4.4 手动扫描时选择多模型链

点击顶部“投毒检测 / 增量审计 / 全量审计”按钮后，在弹窗中可以：

- 分别为审计 / 检查 / 验证三个角色选择模型
- 勾选是否根据审计结果优化对应 skills
- 选择代码平台范围

如果勾选了“根据审计结果优化对应 skills”，且本次为多模型联合校验，则扫描结束后会自动沉淀新的 skills 经验。

### 4.5 不配置 AI 模型

若不配置任何 LLM，系统自动降级为 **Semgrep + OpenSCA** 静态扫描。检测能力有所降低，但不影响基础扫描功能。

## 5. 设置审计计划

每类计划现在都支持：

- 分别设置审计模型、检查模型、验证模型
- 选择是否根据该计划结果优化已有 skills

进入 **系统设置 → 扫描计划** 页面。

### 5.1 三种扫描类型说明

| 类型 | 扫描目标 | 适用场景 | 推荐频率 |
|------|---------|---------|---------|
| **增量投毒扫描** | 所有分支近期提交，专注供应链攻击、后门植入、恶意代码注入 | 高安全要求仓库实时监控 | 每小时 |
| **增量安全审计** | 近期提交的变更文件，上报严重/高危漏洞 | 日常开发流程安全门禁 | 每天（下班后） |
| **全量安全审计** | main/master 主分支完整历史，覆盖所有级别 | 定期安全基线建立 | 每周 |

### 5.2 添加计划

1. 选择扫描类型
2. 选择执行频率：**每小时**（填写分钟数）、**每天**（填写时:分）、**每周**（填写星期和时:分）
3. 可选择绑定特定 **LLM 配置**（留空则使用系统默认）
4. 填写标签名称，方便识别
5. 点击 **保存**

**示例配置**：
```
增量投毒扫描  — 每小时第 0 分     — 标签：实时投毒监控
增量安全审计  — 每天 23:00        — 标签：每日安全审计
全量安全审计  — 每周一 02:00      — 标签：周报安全基线
```

### 5.3 手动触发扫描

在任意页面顶部点击 **投毒检测 / 增量审计 / 全量审计** 按钮，即可打开扫描触发弹窗：

1. 查看本次扫描类型说明
2. 选择本次使用的 AI 配置（可留空，表示使用全局配置）
3. 点击 **开始扫描**

### 5.4 暂停和停止扫描

扫描运行中，顶部状态栏会出现 **暂停** 和 **停止** 按钮：
- **暂停**：扫描进程暂停，可随时点击 **继续** 恢复，不丢失进度
- **停止**：中止本次扫描，本次已发现的漏洞仍会保存

## 6. 配置告警通知

进入 **通知渠道** 页面，点击 **添加渠道**。

### 6.1 钉钉机器人

1. 在钉钉群 → 群设置 → 智能群助手 → 添加机器人 → 自定义 Webhook
2. 安全设置选择 **加签**，复制 Secret
3. 复制 Webhook URL
4. 在平台填写 Webhook URL 和 Secret

### 6.2 飞书机器人

1. 在飞书群 → 设置 → 机器人 → 添加自定义机器人
2. 复制 Webhook 地址
3. 在平台填写 Webhook URL

### 6.3 企业微信群机器人

1. 在企业微信群 → 添加群机器人
2. 复制 Webhook 地址
3. 在平台填写 Webhook URL

### 6.4 邮件（SMTP）

| 字段 | 说明 |
|------|------|
| SMTP 服务器 | 如 `smtp.163.com` / `smtp.gmail.com` |
| 端口 | SSL: 465；STARTTLS: 587 |
| 用户名 | 发件邮箱地址 |
| 密码 | 邮箱密码或应用专用密码 |
| 收件人 | 接收告警的邮箱，多个用逗号分隔 |

### 6.5 测试通知

添加渠道后，点击 **测试** 按钮，验证是否能成功收到消息。

## 7. 查看仪表盘

**仪表盘** 是平台首页，展示：

### 统计卡片
- **近期发现总数**：已完成扫描的累计漏洞数
- **严重 / 高危 / 中危**：按级别聚合显示

### 计划摘要
- 首页会展示三类扫描计划的启用状态和最近一次可推导出的下次执行时间

### 最近扫描
- 展示最近扫描记录
- 可直接查看漏洞、打开 HTML 报告、导出多格式文件或查看执行日志

## 8. 查看扫描历史

进入 **扫描历史** 页面，显示所有扫描记录列表。

### 扫描状态说明

| 状态 | 说明 |
|------|------|
| 运行中 | 扫描正在进行 |
| 已完成 | 扫描正常结束 |
| 已停止 | 手动停止 |
| 已暂停 | 等待继续 |
| 已中断 | 服务重启前未完成的扫描 |
| 错误 | 扫描过程出现异常 |

### 查看漏洞详情

点击某次扫描记录 → **查看漏洞** 按钮，进入漏洞列表。

**漏洞列表功能**：
- 按严重级别（严重/高危/中危/低危）筛选
- 按状态（新增/修复中/已修复/暂不处理）筛选
- 点击漏洞标题查看完整详情

**漏洞详情包含**：
- 漏洞标题与严重级别
- 所在仓库、文件路径、提交 SHA
- 提交作者（代码归属人）
- 漏洞描述：具体风险说明
- 修复建议：针对性修复指导
- 状态操作按钮

## 9. 处理漏洞

每条漏洞支持以下状态流转：

```
新增（new）
  ├── → 修复中（fixing）   : 已指派开发者处理
  ├── → 已修复（fixed）    : 漏洞代码已修改完成
  └── → 暂不处理（wont_fix）: 确认为误报或已知风险可接受
```

### 9.1 标记漏洞状态

在漏洞详情页面，点击底部状态按钮即可更新。

### 9.2 加入白名单

对于确认的误报，请进入 **误报白名单** 页面手动添加规则。规则支持按“仓库 + 文件 + 标题”三项组合匹配，留空字段表示通配。

## 10. 即时检测

**即时检测** 适合在提交前对可疑代码片段快速检查，无需等待计划任务。

### 10.1 粘贴代码检测

1. 进入 **即时分析** 页面
2. 在文本框粘贴代码片段
3. 填写文件名
4. 点击 **开始分析**
5. 检测完成后结果展示在下方

### 10.2 上传文件检测

1. 切换到 **上传文件** 标签
2. 拖放或点击上传代码文件或 **ZIP 压缩包**
3. 点击 **开始检测**

> **提示**：ZIP 文件中会自动过滤图片、编译产物、锁文件等非代码文件。

## 11. 白名单管理

进入左侧导航 **误报白名单** 页面，查看和管理所有白名单条目。

### 白名单匹配规则

| 配置 | 匹配范围 |
|------|---------|
| 仓库 + 文件 + 标题 | 精确匹配该仓库下该文件的该类漏洞 |
| 仓库 + 标题（文件留空） | 匹配该仓库下所有文件的该类漏洞 |
| 仅标题（仓库和文件留空） | 全局匹配所有仓库的该类漏洞（谨慎使用） |

### 删除白名单

在白名单列表中找到条目，点击 **删除** 按钮，删除后该指纹的漏洞将重新被上报。

## 12. 系统设置

进入 **系统设置** 页面。

### 12.1 扫描参数

| 参数 | 说明 | 建议值 |
|------|------|--------|
| 单文件最大 Diff 字符数 | LLM 分析时每个文件的最大输入字符数，超出截断 | 8000~16000 |
### 12.2 工具 Token 配置

OpenSCA 用于检测依赖文件（`package.json`、`pom.xml`、`go.mod` 等）中的已知 CVE 漏洞。

如有 OpenSCA 官方账号，填入 Token 可获得更全面的漏洞库查询能力。

Semgrep Token 为可选项，填写后可启用更多 Pro 规则；不填写时使用社区规则。

### 12.3 产品授权

代码审计系统的系统设置页保留 **产品授权** 卡片，用于客户侧导入和查看授权状态。

| 字段 | 说明 |
|------|------|
| 机器码 | 当前平台实例标识，需提供给授权签发方 |
| 授权状态 | 当前导入授权文件后的校验结果 |
| 上传授权文件 | 上传 `license.json/.lic/.txt`，自动提取并应用授权 |
| 下载机器码文件 | 导出当前实例机器码 JSON，便于交给签发方生成授权 |

> **重要**：当前版本授权校验**强制启用**，不可关闭。未导入有效授权文件时，投毒检测、增量审计、全量审计、即时分析等扫描功能均会返回 403 错误，系统设置和登录功能不受影响。请在首次部署后立即前往系统设置导入授权文件。

导入授权文件后，系统会校验：
- 签名是否有效
- 产品名称是否匹配
- 是否过期
- 若授权绑定机器码，则当前实例是否一致

当授权进入临近到期窗口时，顶部状态区和右下角授权标识会显示到期提醒。

若授权中带有 `features` 列表，启用授权校验后系统会仅放行对应能力：
- `poison_scan`
- `incremental_audit`
- `full_audit`
- `instant_analysis`

### 12.4 Syslog 配置

| 字段 | 说明 |
|------|------|
| 启用 | 是否向外部 Syslog 服务器转发日志 |
| 服务器地址 | Syslog 服务器 IP 或域名 |
| 端口 | 通常为 514（UDP）或自定义 |
| 协议 | UDP / TCP |
| Facility（可多选） | 日志来源类别，支持同时选择多个，如“应用日志（local0）”、“系统服务（daemon）” |

转发的事件类型：登录/登出、扫描启动/完成、漏洞发现、配置变更。

## 14. 应急排查与组件查询

进入左侧导航 **应急排查** 页面，可进行跨仓库快速排查。

### 14.1 恶意依赖应急排查

在“软件成分分析（SCA）”卡片中输入依赖名（如 `event-stream`、`xz-utils`），可执行：
- **模糊检测恶意依赖**：按关键词模糊匹配组件
- **精确检测恶意依赖**：按组件名精确匹配

可附加过滤条件：仓库、代码平台来源（GitHub/GitLab/Gitee/Bitbucket 等）。

### 14.2 组件清单查询

输入组件关键字后点击“查询组件”，可查看：
- 来源平台
- 仓库
- 生态（pypi/npm/maven/gomod/cargo/composer/rubygems）
- 组件名与版本
- 对应依赖清单文件

### 14.3 全代码库关键词检索

用于紧急定位可疑路径/页面/字符串（如 `swagger.html`、`backdoor`）：
1. 输入关键字
2. 选择是否区分大小写
3. 可按仓库、来源过滤
4. 点击“全库检索”

结果会显示命中仓库、文件、行号和命中行片段。

查询结果支持分页展示，可切换每页 20 / 50 / 100 条。

> 说明：该能力基于全量审计同步的 `synced_repos/` 快照，建议先执行一次全量审计以确保数据最新。

## 15. 常见问题

**Q: 添加了代码平台账户但没有拉到任何提交？**

A: 检查以下几点：
1. Token 权限是否足够（GitHub 需要 `repo` 读权限）
2. Owner 填写是否正确（区分大小写，GitHub 组织名与用户名不同）
3. 点击 **测试连接**，查看具体错误信息
4. GitLab 私有化部署需确认自定义域名可以从平台服务器访问

---

**Q: 扫描显示"AI 额度耗尽"？**

A: LLM API Key 的额度已用完或余额不足。解决方式：
1. 前往对应 AI 提供商充值或更换 API Key
2. 在 **AI 配置** 中更新 API Key
3. 或切换到其他可用的 LLM 配置

---

**Q: 收不到钉钉/飞书通知？**

A:
1. 点击渠道右侧的 **测试** 按钮，查看是否报错
2. 钉钉开启了安全签名的机器人，需同时填写 Webhook URL 和 Secret
3. 检查机器人是否还在群内，或是否被禁用
4. 确认服务器网络可以访问钉钉/飞书的 Webhook 域名

---

**Q: 如何生成产品授权文件并上传？**

A:
1. 在目标机器打开 **系统设置 → 产品授权**，点击“下载机器码文件”或复制机器码  
2. 在签发环境准备授权私钥，并设置 `LICENSE_PRIVATE_KEY_PATH`  
3. 执行命令生成授权文件：

```bash
python3 license_manager.py generate-file \
  --customer '客户名称' \
  --expires-at '2027-12-31T23:59:59Z' \
  --machine-code '机器码' \
  --feature poison_scan \
  --feature incremental_audit \
  --feature full_audit \
  --feature instant_analysis \
  --output ./license_customer.json
```

4. 回到系统设置，点击“上传并应用”，选择 `license_customer.json` 即可生效

**Q: 后台授权管理系统在哪？**

A:
1. 后台授权管理系统为公司内部独立部署系统
2. 不随客户侧代码审计平台一同交付
3. 客户侧平台只负责展示机器码、导入授权文件和执行本地验签

---

**Q: 误报太多，能否调整？**

A: 可从以下几个方面优化：
1. **加入白名单**：对确认的误报单条处理，后续自动过滤
2. **更换更强模型**：能力更强的模型（如 GPT-4o、Claude Opus）误报率更低
3. **调整 Prompt**：在 **系统设置 → Prompt 管理** 中修改对应文件类型的 Prompt，增加误报过滤指令
4. **使用增量审计模式**：增量审计默认只上报严重/高危级别，减少中低危噪音

---

**Q: 扫描后状态显示"已中断"？**

A: 这是正常现象，表示服务上次运行期间因重启或异常退出导致扫描未完成。平台在重启时自动将这类扫描标记为"已中断"，不影响已保存的漏洞数据。可手动触发一次新的扫描继续工作。

---

**Q: 如何只审计特定仓库？**

A: 目前代码平台以 Owner（组织/用户）为粒度拉取，会拉取该 Owner 下所有可访问仓库。如需只监控部分仓库，建议：
1. 创建一个专用账户 / Token，只授权目标仓库的访问权限
2. 将该账户配置为独立的代码平台账户

---

**Q: 平台重启后需要重新登录？**

A: 是的，登录 Token 存储在服务进程内存中，服务重启后所有会话失效，需重新登录。这是正常的安全机制。

## 附录：漏洞严重级别说明

| 级别 | 颜色 | 描述 | 典型漏洞 |
|------|------|------|---------|
| 严重（Critical） | 红色 | 可直接造成系统被控、数据泄露 | RCE、SQL 注入、后门植入 |
| 高危（High） | 橙色 | 可造成重要功能被绕过或数据损失 | 认证绕过、IDOR、SSRF |
| 中危（Medium） | 黄色 | 有潜在安全风险，需结合场景评估 | XSS、CSRF、弱加密 |
| 低危（Low） | 绿色 | 信息泄露、代码质量类安全问题 | 堆栈信息暴露、注释中含密钥 |

## 附录：支持的漏洞检测类型

**前端**
XSS、原型链污染、敏感数据泄露（localStorage/sessionStorage）、不安全存储、CSP 配置缺失、CORS 错误配置、postMessage 未校验来源、开放重定向、点击劫持、供应链投毒

**后端**
SQL 注入、NoSQL 注入、命令注入、XXE、认证与授权缺陷（JWT 伪造、IDOR、OAuth2 缺陷）、批量赋值、敏感信息泄露（API Key / 密码硬编码）、弱加密算法、SSRF、文件路径穿越、不安全反序列化、CSRF、业务逻辑漏洞、速率限制缺失

**依赖与供应链**
恶意代码注入、后门植入、typosquatting（包名仿冒）、混淆代码检测、异常外联请求、数据外泄逻辑、已知 CVE（通过 OpenSCA）
